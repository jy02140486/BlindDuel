# 探索 / 战斗 速度拆分方案

## 1. 背景与目标

- 当前 `CharacterBase` 只有一个速度字段 `baseWalkSpeed`（来自 `config.walkSpeed`），被探索与战斗共用。
- 历史计划 `archived/EXPLORE_MOVEMENT_DESIGN.md` §2.3 提出过"探索阶段基础移动速度需要调快 / 战斗阶段保持当前 tuning"，但代码层并未落地 mode 维度速度切换。
- 现状下若要"探索比战斗快"，只能用 `speedMultiplier` buff 在 ExploreMode.enter/exit 临时加减——这种做法语义模糊（buff 应表达临时增益而非模式基线），且会在 buff 列表里留常驻条目。

**目标**：把单一 `walkSpeed` 拆成 `walkSpeed`（探索基线）和 `moveSpeed`（战斗基线）两个独立字段，由 `BattleMode.enter/exit` 切换 `activeSpeedMode`，使 `getEffectiveSpeed()` 自动选源。

## 2. 现状回顾

速度链路三处读取点：

| 位置 | 用途 | 是否需要随 mode 切换 |
|---|---|---|
| [CharacterBase._applyMovement](../scripts/Enties/CharacterBase.js) → `getEffectiveSpeed()` | 实际位移 | ✅ 需要 |
| [AIKnowledgeRegistry.js:77](../scripts/Systems/AIKnowledgeRegistry.js) | 战斗 AI 推断对手移速能力 | ⚠️ 战斗态恒定值，应读战斗速度（待确认） |
| [FollowingBehavior.js:61](../scripts/Systems/NpcBehaviors/FollowingBehavior.js) | 同伴跟随写 `npc.baseWalkSpeed` | ❌ 探索态恒定值，保持原样 |

`BattleMode` 已有现成"进入写 combatant 字段、退出清空" pattern（[BattleMode.js:30-47](../scripts/Systems/Modes/BattleMode.js) 写 `_battleYMin/_battleYMax`），本方案照搬该 pattern 写 `activeSpeedMode`。

## 3. 设计方案

### 3.1 字段定义（`scripts/Enties/CharacterBase.js` 构造函数）

```js
this.baseWalkSpeed = config.walkSpeed ?? 2.4;                 // 探索基线，保留原名
this.baseMoveSpeed = config.moveSpeed ?? this.baseWalkSpeed;  // 战斗基线，新增；fallback 保证旧调用不破
this.activeSpeedMode = "walk";                                // 当前生效模式：walk | move
```

### 3.2 `getEffectiveSpeed()` 选源

只改基础值来源一行，buff / tag 加成逻辑保持不变：

```js
getEffectiveSpeed() {
    let speed = this.activeSpeedMode === "move"
        ? this.baseMoveSpeed
        : this.baseWalkSpeed;
    if (this.hasTag("parryBonus") || this.hasTag("chainBonus")) {
        speed *= this._moveSpeedBonus;
    }
    if (this.buffsProvider && typeof this.buffsProvider.getSpeedMultiplier === "function") {
        speed *= this.buffsProvider.getSpeedMultiplier();
    }
    return speed;
}
```

### 3.3 Mode 切换写入（`scripts/Systems/Modes/BattleMode.js`）

照搬 `_battleYMin` pattern：

```js
enter(payload) {
    // ... 原有逻辑 ...
    for (const combatant of this._combatants ?? []) {
        if (combatant?.setFacingMode) { /* ... */ }
        if (stageBounds && combatant) {
            combatant._battleYMin = stageBounds.minY ?? null;
            combatant._battleYMax = stageBounds.maxY ?? null;
        }
        if (combatant) combatant.activeSpeedMode = "move";   // 新增
    }
}

exit() {
    for (const combatant of this._combatants ?? []) {
        if (combatant) {
            combatant._battleYMin = null;
            combatant._battleYMax = null;
            combatant.activeSpeedMode = "walk";               // 新增
        }
    }
}
```

**ExploreMode.enter 防御性复位**——正常路径下 BattleMode.exit 已写回 `"walk"`；为防异常退出路径（exit 被跳过），ExploreMode.enter 兜底再写一次 `"walk"`，若发现进入时不是 `"walk"` 则 `console.warn` 提示。

### 3.4 工厂数据（`scripts/CharacterFactory.js`）

```js
const DEFAULT_CHARACTER_OPTIONS = {
    pxToWorld: 0.03,
    collisionThicknessPx: 40,
    moveDeadzone: 0.2,
    walkSpeed: 1.4,    // 探索：快（跑图）
    moveSpeed: 0.4,    // 战斗：慢（走位/防穿模）

```

NPC 装配路径（[CharacterFactory.js:438](../scripts/CharacterFactory.js)）**不**加 `moveSpeed`——NPC `capabilities.combat = false`，永不进战斗态，fallback 到 walkSpeed 即可。

### 3.5 AIKnowledgeRegistry 读源切换 + 字段重命名（已确认）

[AIKnowledgeRegistry.js](../scripts/Systems/AIKnowledgeRegistry.js)：

- 值源：`character.baseMoveSpeed ?? character.baseWalkSpeed ?? 0`（取战斗态值）
- 变量名 + 输出字段名：`walkSpeed` → `moveSpeed`（语义对齐：存的是战斗移动速度）

```js
const moveSpeed = character.baseMoveSpeed ?? character.baseWalkSpeed ?? 0;
// ...
return {
    characterId: character.id,
    pxToWorld,
    moveSpeed,
    attacks: attackProfiles,
    movement: { moveSpeed, stateDisplacements }
};
```

消费者影响：AIController 只读 `kbProfile.attacks`，不读速度字段，零影响。

## 5. 边界与注意点

1. **NPC 安全**：NPC 不配 `moveSpeed` → fallback 到 `walkSpeed`；且 NPC 永不进 BattleMode，`activeSpeedMode` 永远是 `"walk"`。
2. **Sequencer 期间**：`controlledBySequence === true` 时 `_applyMovement` 开头就 return（已有守卫），速度字段不参与，无影响。
3. **切换瞬间**：`BattleMode.enter()` 是同步调用，下一帧 `fixedUpdate` 即生效。enter 发生在 sequence 编排之后——sequence 期间角色已经 `controlledBySequence=true`，所以没有"半路用错速度位移"的问题。
4. **不与 `moveSpeedBonus` 混淆**：`moveSpeedBonus` 是状态内**乘法加成**（如 parry 状态加速），跟"战斗模式基础速度"是两层概念。本方案不碰它。
5. **Debug panel** 显示的是 `currentSpd`（实际位移速度），无需改。

## 6. 不做的事（防止 over-engineering）

- ❌ 不在 stateGraph JSON 里加 `moveSpeed`（命名容易与 `moveSpeedBonus` 混淆）
- ❌ 不删 `baseWalkSpeed`（保持向后兼容）
- ❌ 不引入 `SpeedProfile` / `getSpeedForMode(mode)` 这类抽象（单用例不需要）
- ❌ 不动 buff 机制

## 7. 已确认决定

1. **数值**：`walkSpeed: 1.4`（探索，快），`moveSpeed: 0.4`（战斗，慢）。探索快于战斗，方便跑图；战斗慢下来便于走位。
2. **AIKnowledgeRegistry 读源切换**：✅ 已做。值源从 `baseWalkSpeed` 切到 `baseMoveSpeed`，且字段名 `walkSpeed`→`moveSpeed` 统一命名。
3. **ExploreMode.enter 防御性复位**：✅ 已加。正常路径 BattleMode.exit 已复位；异常路径（exit 被跳过）由 ExploreMode.enter 兜底 + `console.warn` 提示。
4. **方法改名**：`getEffectiveWalkSpeed()` → `getEffectiveSpeed()`，消除"Walk"命名歧义（该方法按 mode 返回 walk 或 move 速度）。

## 8. 验收标准

- [ ] 探索态位移速度 = `walkSpeed(1.4) × buff × tagBonus`
- [ ] 进入战斗后位移速度 = `moveSpeed(0.4) × buff × tagBonus`
- [ ] 退出战斗回到探索，速度自动回到 `walkSpeed`
- [ ] NPC 跟随速度与改前一致（`FollowingBehavior` 路径不变）
- [ ] AI 知识注册表输出的 `moveSpeed` 字段反映战斗态值（0.4）
- [ ] 切换瞬间无位置抖动（依赖 `controlledBySequence` 守卫，已存在）
- [ ] 异常路径下 ExploreMode.enter 的 `console.warn` 能触发（手动测试：跳过 BattleMode.exit 直接到 ExploreMode）
