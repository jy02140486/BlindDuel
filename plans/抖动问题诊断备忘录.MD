# 26.7.19 抖动问题诊断备忘录

## 用途

跟随相机下人物抖动**偶发重现**（非必现）时使用本备忘录。
未来重现时按本文档的步骤启用诊断日志，捕捉数据贴出来分析。

## 诊断日志体系

### 已埋点的日志（正常运行时不会刷屏，仅在异常时输出）

| 前缀 | 位置 | 触发条件 | 用途 |
|------|------|---------|------|
| `[JITTER_DETECTED]` | CharacterBase._checkJitter | 单帧位移 > 0.2，或 5 帧内方向反转 ≥ 2 次 | 角色位置抖动 |
| `[JITTER_CONTEXT]` | 同上 | 同上（紧跟 DETECTED） | 上下文：actor 状态、moveIntent、相机位置/目标 |
| `[JITTER_HISTORY]` | 同上 | 同上（仅 OSCILLATION 类型） | 最近 5 帧位置变化方向 |
| `[CAM_JITTER_DETECTED]` | CameraManager.#checkCameraJitter | 相机位置/目标突变 > 0.3 | 相机抖动 |
| `[CAM_JITTER_POS]` | 同上 | 同上（紧跟 DETECTED） | 相机位置 delta 详情 |
| `[CAM_JITTER_TARGET]` | 同上 | 同上 | 相机 target delta 详情 |
| `[CAM_JITTER_BLEND]` | 同上 | 同上 + blend 激活时 | blend 状态详情 |
| `[TimelineSeq] moveActorTo WARN` | TimelineSequencer.moveActorTo.end | snap 距离 > 0.1 | sequencer 结束时的位置修正警告 |

### 节流策略

所有日志都有节流（200-3000ms），避免 tick 级日志淹没控制台。但 `[JITTER_DETECTED]` 一旦触发说明真异常，必看。

## 已知抖动风险点（按优先级）

### 1. sequencer 期间的位置双写（已修复，仍是首要嫌疑）

**历史**：见 [26.7.11 ScriptedCamera follow + moveActorTo 位置双写修复](./26.7.11%20ScriptedCamera%20follow%20+%20moveActorTo%20位置双写修复.MD)

**机制**：sequencer 的 `moveActorTo` clip 写绝对位置，同时 CharacterBase.fixedUpdate 的 `_applyMovement` 写增量位移。两者叠加导致位置异常。

**守卫**：`CharacterBase._applyMovement` 开头有 `if (this.controlledBySequence) { this.currentSpd = 0; return; }`

**重现验证**：
- 看 `[JITTER_DETECTED]` 的 `controlledBySequence` 字段是否为 `true`
- 看 `[TimelineSeq] moveActorTo WARN` 是否在抖动瞬间附近出现（snap 距离过大）
- 看 activeClips 中是否有 `moveActorTo`

**如果重现且 controlledBySequence=true**：说明守卫失效或有新的双写路径。
**如果重现且 controlledBySequence=false**：说明抖动来自非 sequencer 路径，看下条。

### 2. ExploreMode.updateRender 的 z = y * Z_FACTOR

**机制**：`entity.root.position.z = entity.root.position.y * Z_FACTOR`。y 突变会同步传播到 z，可能造成 sprite 互相穿透。

**当前状态**：z 振荡检测已被移除（清理 prop 闪烁诊断时一起清了）。

**重现验证**：如果未来需要重新检测 z 振荡，参考 commit `26.7.19` 的清理 diff，恢复 `_zPosHistory` 跟踪和 `[JDBG:Z]` 日志。判断标准：5 帧内 z 方向反转 ≥ 2 次。

### 3. 相机 blend 期间的 from/to state 差值

**机制**：CameraManager.startBlend 从 fromState lerp 到 toState。如果两个 state 差值过大，blend 中段可能出现位置突变。

**重现验证**：
- 看 `[CAM_JITTER_DETECTED]` 的 `blending` 字段是否为 `true`
- 看 `[CAM_JITTER_BLEND]` 的 elapsed/duration/toRigId
- 如果 blending=true 且 toState.pos 与当前 pos 差值大 → blend 配置问题

### 4. follow 相机的 lerp k 系数

**机制**：ScriptedCameraRig 的 follow 模式用 `k = 1 - Math.pow(1 - lerp, dtMs / 16.67)` 帧率无关 lerp。dtMs 异常（卡顿后第一帧）会让 k 异常。

**重现验证**：
- 看 `[JITTER_DETECTED]` 是否伴随卡顿（dtMs 突然变大）
- 看 `[CAM_JITTER_DETECTED]` 的 seqBusy 字段，是否 sequencer 忙时触发

## 重现时的标准排查流程

### Step 1：打开控制台过滤
过滤词：JITTER_ 或 CAM_JITTER_ 或 TimelineSeq

正常游戏不会有这些日志。一旦抖动发生，对应日志会被触发（受节流影响可能延迟 3 秒）。

### Step 2：确认抖动类型

- **画面整体抖动（相机抖）**：先看 `[CAM_JITTER_DETECTED]`
- **只有角色位置抖（相机不动）**：先看 `[JITTER_DETECTED]`
- **抖动伴随 sequencer 启动/结束**：看 `[TimelineSeq] moveActorTo WARN`

### Step 3：贴日志分析

把抖动前后 3-5 秒的日志都贴出来，重点字段：
[JITTER_DETECTED] actor=hero tick=1234 type=OSCILLATION posDelta=(0.002,-0.003) dist=0.004
controlledBySequence=false state=walk moveIntent=(0.5,0) speedMode=walk currentSpd=1.20 reversals=3

分析方向：
1. `controlledBySequence=true` → sequencer 路径，看 moveActorTo / snap
2. `type=DELTA` + `dist` 大 → 单帧位置突变，看 sequencer snap 或外部写入
3. `type=OSCILLATION` + `reversals` 高 → 小幅来回抖，看 follow lerp 或 _applyMovement
4. `moveIntent=(0,0)` 但仍抖 → 非输入驱动，sequencer 或外部
5. `camPos`/`camTarget` 突变 → 相机问题，看 `[CAM_JITTER_*]`

### Step 4：需要时恢复被清理的诊断日志

prop 闪烁诊断日志（`[JDBG:*]`）已被清理。如果怀疑抖动与 prop 相关（如 sequencer 同时驱动 prop 和角色），可参考 commit `26.7.19` 的清理 diff 反向恢复以下检测：

- `[JDBG:STENCIL]` stencil 状态翻转
- `[JDBG:Z]` z 振荡
- `[JDBG:ORDER]` alphaIndex 顺序翻转

但根据当前理解，prop 闪烁问题已通过预加载纹理彻底修复，这些检测应不再需要。

## 待办（未来如重现且日志不足时）

1. 如果 `[JITTER_DETECTED]` 没触发但视觉仍抖 → 降低 `_jitterThreshold`（当前 0.2）和 `_oscillationThreshold`（当前 2）
2. 如果怀疑是 follow lerp 异常 → 在 ScriptedCameraRig.compute 加日志输出 k 值和 dtMs
3. 如果怀疑是 renderloop 时序 → 在 Scene.updateRender 加帧时间日志
4. 如果怀疑是 z 穿透 → 恢复 `[JDBG:Z]` 检测
